## 1. AgentPit OAuth2 单点登录实现

### 1.1 问题背景

网站 `https://destiny.agentpit.io/login` 的「agentpit 授权登陆」按钮跳转到了错误的授权流程（Second.me OAuth），需要改为正确的 AgentPit OAuth2 授权。

**OAuth2 凭证：**

| 参数 | 值 |
|------|-----|
| Client ID | `cmn5vk3l1001560777k8fb1vh` |
| Client Secret | `cmn5vk3l100166077koz4m9ed` |
| Callback URL | `https://destiny.agentpit.io/api/auth/agentpit/callback` |

### 1.2 实现方案

采用**弹窗式 OAuth2 授权码流程**：

```
用户点击按钮
    → 打开弹窗 /api/auth/agentpit
    → 重定向至 AgentPit 授权页
    → 用户授权后回调 /api/auth/agentpit/callback
    → 后端交换 Token、获取用户信息、生成 JWT
    → 通过 window.opener.postMessage 通知主窗口
    → 主窗口写入 localStorage，完成登录
```

**AgentPit OAuth 接口地址：**

| 接口 | 地址 |
|------|------|
| 授权页 | `https://www.agentpit.io/api/oauth/authorize` |
| Token 交换 | `https://www.agentpit.io/api/oauth/token` |
| 用户信息 | `https://www.agentpit.io/api/oauth/userinfo` |

### 1.3 关键文件变更

#### 数据库迁移
`database/add_oauth_support.sql`
- `tb_user` 表新增 `oauth_provider`（VARCHAR 50）、`oauth_id`（VARCHAR 200）字段
- `password` 字段改为可空（OAuth 用户无本地密码）
- 新增唯一索引 `uk_oauth(oauth_provider, oauth_id)`

#### 实体 & Mapper
- `entity/User.java`：新增 `oauthProvider`、`oauthId` 字段
- `mapper/UserMapper.java`：新增 `findByOauthId`、`insertOauthUser`、`updateOauthUserInfo`

#### 核心服务
`service/AgentpitOAuthService.java`（全新）
- **全响应式**：所有 HTTP 调用均用 `WebClient` + `Mono` 链，无 `.block()` 调用
- 数据库操作通过 `Mono.fromCallable().subscribeOn(Schedulers.boundedElastic())` 隔离到独立线程池
- 内嵌 `OAuthResult` DTO 封装结果

#### 控制器
`controller/AgentpitOAuthController.java`（全新）
- `GET /api/auth/agentpit`：302 重定向到 AgentPit 授权页（供弹窗调用）
- `GET /api/auth/agentpit/callback`：处理回调，返回含 `postMessage` 脚本的 HTML 页

#### 安全配置
`config/SecurityConfig.java`
- `PUBLIC_API_PATTERNS` 和 `PUBLIC_API_PREFIXES` 均加入 `/api/auth/agentpit/**`，允许未认证访问

#### 前端
`src-frontend/pages/LoginPage.jsx`
- 弹窗式授权：`window.open('/api/auth/agentpit', ...)`
- 监听 `message` 事件接收登录结果，写入 `localStorage`

#### 配置文件
`src/main/resources/application.yml`：新增 `agentpit.oauth.*` 配置块，通过环境变量注入

### 1.4 踩坑与修复

#### 问题一：硬编码旧 OAuth 地址
- **现象**：`.env` 已更新为 AgentPit URL，但授权仍跳到 Second.me
- **原因**：JAR 包内的 `application.yml` 仍写死了旧地址
- **修复**：在服务器上用 Maven Docker 容器重新编译 JAR，`docker cp` 替换容器内文件

#### 问题二：WebFlux 响应式死锁
- **现象**：日志显示「获取 access_token 失败」，实际 Token 已拿到但返回 null
- **原因**：`AgentpitOAuthService` 在 WebFlux reactor 线程上调用了 `WebClient.block()`，造成死锁
- **修复**：将全部逻辑改为响应式 `Mono` 链，数据库操作用 `Schedulers.boundedElastic()` 隔离

#### 问题三：Spring Cache `sync` + `unless` 不兼容
- **现象**：`/api/daily-fortune-detail/today` 返回 400 Bad Request
- **原因**：`DailyFortuneDetailService` 的 `@Cacheable` 同时使用了 `sync=true` 和 `unless` 属性，Spring 不支持此组合
- **修复**：删除 `unless` 属性，改用 `condition` 控制缓存条件

#### 问题四：`docker compose restart` 不重读 `.env`
- **现象**：更新 `.env` 后重启容器，环境变量仍是旧值
- **原因**：`restart` 复用旧的容器配置，不重新读取 `.env`
- **修复**：改用 `docker compose up -d --force-recreate --no-deps --build backend`

## 2. 大模型 API 从 DeepSeek 切换至 Gemini

### 2.1 问题背景

AI 解读功能（八字、六爻、紫微等）报错：

```
DeepSeek API Key 未配置，请检查 DEEPSEEK_API_KEY 或 application.yml。
```

**根因**：服务器 `.env` 中 `DEEPSEEK_API_KEY=`（空值），覆盖了 `application.yml` 中的默认 Key。

用户要求将大模型切换为 Gemini 代理（`https://gemini.agentpit.io/`）：

| 参数 | 值 |
|------|-----|
| API Base URL | `https://gemini.agentpit.io/v1` |
| API Key | `sk-iOIqwYXa7QKkQpOe3315B0F9E79344Fd98Ae6e12B44a3902` |
| 默认模型 | `gemini-2.0-flash` |

> Gemini 代理兼容 OpenAI Chat Completions 接口格式，DeepSeek 的请求代码无需修改。

### 2.2 修改方案

Gemini 代理使用与 DeepSeek 相同的 OpenAI 兼容格式（`POST /v1/chat/completions`），因此只需更新配置，无需改动请求逻辑。

### 2.3 关键文件变更

#### `src/main/resources/application.yml`

```yaml
# 修改前
deepseek:
  api:
    key: ${DEEPSEEK_API_KEY:sk-370d7060082b4d02a4710f8127e11a47}
    endpoint: ${DEEPSEEK_API_ENDPOINT:https://api.deepseek.com/v1/chat/completions}
  model: deepseek-chat

gemini:
  api:
    base-url: ${ONE_API_BASE_URL:${GEMINI_API_BASE_URL:http://104.197.139.51:3000/v1}}

# 修改后
deepseek:
  api:
    key: ${DEEPSEEK_API_KEY:${ONE_API_KEY:sk-iOIqwYXa7QKkQpOe3315B0F9E79344Fd98Ae6e12B44a3902}}
    endpoint: ${DEEPSEEK_API_ENDPOINT:https://gemini.agentpit.io/v1/chat/completions}
  model: ${DEEPSEEK_MODEL:gemini-2.0-flash}

gemini:
  api:
    base-url: ${ONE_API_BASE_URL:${GEMINI_API_BASE_URL:https://gemini.agentpit.io/v1}}
```

`scene-image.api.base-url` 默认值同步更新为 `https://gemini.agentpit.io/v1`。

#### `bazi/service/BaZiDeepSeekService.java`

- 新增 `@Value("${deepseek.model:gemini-2.0-flash}") private String model;`
- 删除两处硬编码 `"deepseek-chat"`，改用 `model` 字段
- 错误提示改为通用措辞（不再写死"DeepSeek"）

### 2.4 服务器部署步骤

> 由于 Dockerfile 使用 `COPY target/*.jar /app/app.jar`，JAR 打包进镜像，
> `docker compose restart` 不会更新 JAR 和环境变量，需走完整重建流程。

```bash
# 1. 上传新源码
scp -i ~/.ssh/id_rsa_google_longterm /tmp/baziback-src.tar.gz a1@35.198.208.41:/tmp/

# 2. 在服务器上解压 + 编译
cd /opt/baziback-docker/app
tar xzf /tmp/baziback-src.tar.gz
docker run --rm -v "$(pwd)":/workspace -w /workspace \
  maven:3.9-eclipse-temurin-17 mvn clean package -DskipTests -q

# 3. 更新 .env（关键！）
# DEEPSEEK_API_KEY=sk-iOIqwYXa7QKkQpOe3315B0F9E79344Fd98Ae6e12B44a3902
# DEEPSEEK_API_ENDPOINT=https://gemini.agentpit.io/v1/chat/completions
# ONE_API_KEY=sk-iOIqwYXa7QKkQpOe3315B0F9E79344Fd98Ae6e12B44a3902
# ONE_API_BASE_URL=https://gemini.agentpit.io/v1
# GEMINI_API_KEY=sk-iOIqwYXa7QKkQpOe3315B0F9E79344Fd98Ae6e12B44a3902
# GEMINI_API_BASE_URL=https://gemini.agentpit.io/v1

# 4. 强制重建容器（同时重读 .env 和新 JAR）
docker compose up -d --force-recreate --no-deps --build backend
```

**关键注意**：`docker compose restart` 仅重启进程，**不重读** `.env` 也**不更新** JAR。
必须使用 `--force-recreate --build` 才能使两者生效。

## 3. 遗留问题与注意事项

### 场景图生成（待确认）

根据 One-API Gemini 适配器修复文档（2026-03-23），图片生成模型需满足：

1. 请求中设置 `generationConfig.responseModalities: ["TEXT", "IMAGE"]`
2. One-API 响应转换需支持 `inlineData` → `image_url` 格式

当前后端 `GeminiService` 调用图片生成时，需确认响应解析已支持多模态 content 数组格式：
```json
{
  "content": [
    {"type": "text", "text": "描述"},
    {"type": "image_url", "image_url": {"url": "data:image/jpeg;base64,..."}}
  ]
}
```

**推荐图片生成模型**：`gemini-3-pro-image-preview`（已验证可用）

### `docker compose restart` vs `--force-recreate` 区别

| 操作 | 读取新 `.env` | 更新镜像/JAR |
|------|------------|------------|
| `restart` | ❌ | ❌ |
| `up -d` | ❌（容器已存在时） | ❌ |
| `up -d --force-recreate` | ✅ | ❌（不重 build） |
| `up -d --force-recreate --build` | ✅ | ✅ |

**结论**：每次更新 `.env` 或 JAR 后，必须使用 `--force-recreate --build`。
