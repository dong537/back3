## 2. 整体架构

```
互联网
  │
  ▼ 443/80
Caddy（HTTPS 反向代理）
  │  destiny.agentpit.io → 127.0.0.1:3000
  ▼
frontend 容器（Nginx，端口 3000→8080）
  │  /api/* → backend:8088
  ▼
backend 容器（Spring Boot WebFlux，端口 8088）
  │
  ▼
MySQL（宿主机 MariaDB，端口 3306）
```

**端口占用一览：**

| 端口 | 服务 | 说明 |
|------|------|------|
| 80 / 443 | Caddy | 对外 HTTP/HTTPS |
| 3000 | frontend 容器 | Caddy 反代目标 |
| 8088 | backend 容器 | Spring Boot |
| 3306 | 宿主机 MySQL | 供 backend 容器访问 |
| 3307 | mysql 容器（可选） | 独立 MariaDB，仅 local-db profile 启动 |

> **注意**：当前生产环境数据库运行在**宿主机 MySQL**（非 Docker），backend 通过 `host.docker.internal:3306` 访问。

## 4. 首次全量部署

### 4.1 前置条件

```bash
# 在服务器上安装 Docker
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER

# 安装 Caddy（HTTPS 反向代理）
sudo apt install -y caddy

# 创建应用目录
sudo mkdir -p /opt/baziback-docker/app
sudo chown -R $USER:$USER /opt/baziback-docker
```

### 4.2 上传代码

```bash
# 本地：打包源码（排除 node_modules、target 等大目录）
cd /path/to/baziback
tar czf /tmp/baziback-full.tar.gz \
  src/ src-frontend/ database/ sql/ docker/ \
  pom.xml package.json package-lock.json \
  docker-compose.yml vite.config.js tailwind.config.js postcss.config.js \
  index.html public/

# 上传到服务器
scp -i ~/.ssh/id_rsa_google_longterm /tmp/baziback-full.tar.gz a1@35.198.208.41:/opt/baziback-docker/app/

# 在服务器上解压
ssh -i ~/.ssh/id_rsa_google_longterm a1@35.198.208.41
cd /opt/baziback-docker/app
tar xzf baziback-full.tar.gz
```

### 4.3 配置 .env

```bash
cd /opt/baziback-docker/app
cp env.example .env
# 编辑 .env，填写实际配置
nano .env
```

参考第 7 节的完整 `.env` 模板。

### 4.4 构建并启动容器

```bash
cd /opt/baziback-docker/app

# 第一步：用 Maven Docker 容器编译后端 JAR
docker run --rm \
  -v "$(pwd)":/workspace \
  -w /workspace \
  maven:3.9-eclipse-temurin-17 \
  mvn clean package -DskipTests -q

# 第二步：构建镜像并启动所有服务
docker compose up -d --build

# 查看启动状态
docker compose ps
docker compose logs -f --tail=50
```

### 4.5 初始化数据库

> 若使用宿主机 MySQL，需提前创建数据库和用户：

```sql
CREATE DATABASE IF NOT EXISTS bazi CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER IF NOT EXISTS 'bazi'@'%' IDENTIFIED BY 'bazi123456';
GRANT ALL PRIVILEGES ON bazi.* TO 'bazi'@'%';
FLUSH PRIVILEGES;
```

> 若使用 Docker 内置 MariaDB（`local-db` profile），初始化 SQL 会在首次启动时自动执行：

```bash
docker compose --profile local-db up -d
```

**手动执行增量 SQL（首次或新增表时）：**

```bash
# 进入宿主机 MySQL（或 Docker 内的 MariaDB）
mysql -h 127.0.0.1 -P 3306 -u bazi -pbazi123456 bazi

# 执行 SQL 文件
mysql -h 127.0.0.1 -P 3306 -u bazi -pbazi123456 bazi < database/add_oauth_support.sql
```

**配置 Caddy HTTPS：**

```bash
# 编辑 /etc/caddy/Caddyfile
cat > /etc/caddy/Caddyfile << 'EOF'
destiny.agentpit.io {
    encode gzip zstd
    reverse_proxy 127.0.0.1:3000
}
EOF

sudo systemctl reload caddy
```

## 5. 增量更新部署（补丁）

### 5.1 只更新后端 JAR

适用场景：只修改了 Java 源码或 `application.yml`，`.env` 无变化。

```bash
# ===== 本地操作 =====
# 打包修改的源文件
cd /path/to/baziback
tar czf /tmp/baziback-src.tar.gz \
  src/main/java \
  src/main/resources \
  pom.xml

# 上传到服务器
scp -i ~/.ssh/id_rsa_google_longterm /tmp/baziback-src.tar.gz a1@35.198.208.41:/tmp/

# ===== 服务器操作 =====
ssh -i ~/.ssh/id_rsa_google_longterm a1@35.198.208.41
cd /opt/baziback-docker/app

# 解压源码
tar xzf /tmp/baziback-src.tar.gz

# 用 Maven Docker 容器编译（约 1-2 分钟）
docker run --rm \
  -v "$(pwd)":/workspace \
  -w /workspace \
  maven:3.9-eclipse-temurin-17 \
  mvn clean package -DskipTests -q

# ⚠️ 关键步骤：docker cp 替换容器内 JAR（restart 不会读新 JAR）
CONTAINER=$(docker compose ps -q backend)
docker cp target/bazi-0.0.1-SNAPSHOT.jar ${CONTAINER}:/app/app.jar

# 重启后端容器
docker compose restart backend

# 验证
docker compose ps backend
docker compose logs backend --tail=30
```

> **为什么不直接 `docker compose restart`？**
> `Dockerfile.backend` 使用 `COPY target/*.jar /app/app.jar` 将 JAR 打入镜像。
> `restart` 只重启进程，不重建镜像，新 JAR 不会生效。
> 使用 `docker cp` 可在不重建镜像的情况下替换运行中容器的文件。

### 5.2 只更新前端静态文件

适用场景：只修改了 React 前端代码。

```bash
# 方法一：重建前端镜像（推荐，彻底干净）
cd /opt/baziback-docker/app
docker compose up -d --force-recreate --no-deps --build frontend

# 方法二：本地构建后上传 dist 目录
# （本地需要 Node.js 环境）
npm run build
scp -i ~/.ssh/id_rsa_google_longterm -r dist/ a1@35.198.208.41:/tmp/dist-new/
# 然后 docker cp 替换容器内静态文件
CONTAINER=$(docker compose ps -q frontend)
docker cp /tmp/dist-new/. ${CONTAINER}:/usr/share/nginx/html/
docker compose restart frontend
```

### 5.3 更新环境变量（.env）

> **重要**：`.env` 更新后，必须使用 `--force-recreate` 重建容器，`restart` 不会读取新值。

```bash
cd /opt/baziback-docker/app

# 编辑 .env
nano .env

# 强制重建受影响的容器（读取新 .env + 新 JAR）
docker compose up -d --force-recreate --no-deps --build backend

# 验证容器内环境变量
docker exec app-backend-1 env | grep -E "DEEPSEEK|ONE_API|GEMINI|AGENTPIT"
```

| 操作 | 读取新 `.env` | 更新 JAR | 说明 |
|------|:---:|:---:|------|
| `docker compose restart` | ❌ | ❌ | 仅重启进程 |
| `docker compose up -d` | ❌ | ❌ | 容器存在时不重建 |
| `docker compose up -d --force-recreate` | ✅ | ❌ | 重建容器但不重构镜像 |
| `docker compose up -d --force-recreate --build` | ✅ | ✅ | 重构镜像 + 重建容器 |
| `docker cp` + `restart` | ❌ | ✅ | 仅更新文件，不重读 .env |

## 8. Nginx / Caddy 配置

### Caddy（宿主机，处理 HTTPS）

文件路径：`/etc/caddy/Caddyfile`

```caddyfile
destiny.agentpit.io {
    encode gzip zstd
    reverse_proxy 127.0.0.1:3000
}
```

- 自动申请并续期 Let's Encrypt TLS 证书
- 将 HTTPS 流量转发至前端容器（端口 3000）

```bash
# 重载 Caddy 配置
sudo systemctl reload caddy

# 查看 Caddy 状态
sudo systemctl status caddy
sudo journalctl -u caddy -f
```

### 前端容器 Nginx（处理静态文件 + API 反代）

文件路径：`docker/nginx/default.conf`

```nginx
server {
    listen 8080;

    # 静态资源（30天缓存）
    location ~* \.(js|css|html|svg|png|jpg|jpeg|gif|ico|woff|woff2)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
        try_files $uri =404;
    }

    # 前端路由（SPA 模式）
    location / {
        try_files $uri $uri/ /index.html;
    }

    # AI 推理流式接口（长连接）
    location /api/deepseek/reasoning-stream {
        proxy_pass http://backend:8088;
        proxy_buffering off;
        proxy_read_timeout 3600s;
        # ...
    }

    # 其他 API 请求
    location /api/ {
        proxy_pass http://backend:8088;
        client_max_body_size 10m;
        proxy_read_timeout 60s;
        # ...
    }
}
```

## 11. 常见问题排查

### 问题 1：更新 .env 后环境变量未生效

**现象**：修改 `.env` 并重启后，`docker exec ... env` 显示仍是旧值。

**原因**：`docker compose restart` 不重读 `.env`，复用旧的容器配置。

**解决**：
```bash
docker compose up -d --force-recreate --no-deps --build backend
# 验证
docker exec app-backend-1 env | grep DEEPSEEK
```

---

### 问题 2：部署新 JAR 后功能未更新

**现象**：上传了新代码并重新编译，但 API 行为没有变化。

**原因**：`Dockerfile.backend` 将 JAR 打入镜像，`restart` 不更新镜像内的文件。

**解决**：
```bash
# 方法一：docker cp（快速，不需要重建镜像）
CONTAINER=$(docker compose ps -q backend)
docker cp target/bazi-0.0.1-SNAPSHOT.jar ${CONTAINER}:/app/app.jar
docker compose restart backend

# 方法二：完整重建（推荐，保证干净）
docker compose up -d --force-recreate --no-deps --build backend
```

---

### 问题 3：AI 解读报错「API Key 未配置」

**现象**：调用 `/api/deepseek/generate-report` 返回 500，提示 Key 未配置。

**原因**：`.env` 中 `DEEPSEEK_API_KEY=`（空值）覆盖了 `application.yml` 中的默认值。

**解决**：
```bash
# 检查容器内实际值
docker exec app-backend-1 env | grep DEEPSEEK_API_KEY

# 更新 .env
nano /opt/baziback-docker/app/.env
# 设置：DEEPSEEK_API_KEY=sk-iOIqwYXa7QKkQpOe3315B0F9E79344Fd98Ae6e12B44a3902

# 重建容器
docker compose up -d --force-recreate --no-deps --build backend
```
